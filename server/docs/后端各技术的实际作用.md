# 后端各技术的实际作用 

● 基于源码分析，下面是后端各技术在项目中的实际作用：

  ---
  1. NestJS 11 — 应用框架

  整个后端的核心框架，提供 模块化架构 + 依赖注入 + 装饰器
  能力。项目中所有控制器（Controller）、服务（Service）、模块（Module）都是
  NestJS 的核心概念。

  - @Module / @Controller / @Injectable — 组织代码为模块，声明依赖关系
  - 全局管道/过滤器/拦截器 — 统一验证、异常处理、日志记录

  2. TypeScript — 开发语言

  全项目使用 TypeScript 编写，提供类型安全。NestJS 的装饰器和 DI 容器天然依赖 TS
  的 emitDecoratorMetadata 元数据反射机制。

  3. TypeORM 1.0 + MySQL — 数据持久化

  用 实体（Entity）+ 仓库（Repository） 模式操作 MySQL 数据库。

  - BaseEntity 定义了 id、createdAt、updatedAt 公共字段
  - SessionEntity / ConversationMessageEntity / EmployeeEntity
  等业务实体映射数据库表
  - DatabaseModule 通过 forRootAsync 异步配置连接（连接池 20，生产环境
  synchronize: false）
  - SessionService 用 @InjectRepository 注入 Repository 进行 CRUD

  4. @nestjs/config — 配置管理

  通过 configuration.ts 将所有环境变量（数据库、Redis、LLM API
  Key、端口等）集中加载为强类型配置，通过 ConfigService 在任意 Service
  中注入使用。

  5. ioredis — Redis 缓存

  RedisService 封装了 Redis 连接和基本操作，在业务层面提供两层缓存：

  - 员工信息缓存 employee:{id}，TTL 5 分钟
  - Agent 响应缓存 agent:{questionHash}，TTL 60 秒（相同问题快速返回）

  6. Vercel AI SDK (ai v6) + @ai-sdk/openai-compatible — AI Agent 核心

  这是整个项目的核心业务引擎。

  - AgentService 使用 generateText() 调用 LLM（默认智谱 GLM-4-Flash，也支持
  Anthropic）
  - StreamingController 使用 streamText() 实现 SSE 流式输出
  - 工具调用（Function Calling）：LLM 可以自主调用定义的工具来获取数据，最多 6
  轮迭代

  7. Zod v4 — 工具参数 Schema 校验

  在 ToolRegistryService 中，每个 AI 工具的输入参数都用 Zod 定义 Schema。LLM
  收到的工具定义里包含这些 Schema，可以生成正确格式的参数，SDK 也会自动校验。

  例如 query_by_structure 工具要求 conditions_json: z.string()，LLM 就知道要传
  JSON 字符串。

  8. class-validator + class-transformer — HTTP 请求参数校验

  在 main.ts 中注册了全局 ValidationPipe：

  - transform: true — 自动将请求体转换为 DTO 实例
  - whitelist: true — 自动剔除 DTO 中未声明的字段

  例如 QueryRequest DTO 用 @IsString()、@MinLength(1)、@MaxLength(500)
  校验用户查询输入。

  9. @nestjs/swagger — API 文档

  在 main.ts 中通过 SwaggerModule.createDocument() 生成 OpenAPI 规范，提供
  Swagger UI 页面访问 /api/docs。DTO 上的 @ApiProperty()
  装饰器让字段带有中文描述和示例值。

  10. Winston + nest-winston — 日志系统

  项目中自定义了 Logger 实例（new Logger(ClassName.name)），底层基于
  Winston。LoggingInterceptor 记录每个 HTTP 请求的耗时。

  11. RxJS (rxjs) — 响应式编程

  - LoggingInterceptor 使用 tap 操作符在 Observable 管道中记录日志
  - StreamingController 使用 Observable 实现 SSE 流式响应（@Sse 装饰器）
  - 返回 Observable<T> 让 NestJS 自动将数据流推送到客户端

  12. 全局异常过滤器 (自定义) — 统一错误处理

  HttpExceptionFilter 统一捕获所有异常，分三种情况返回统一格式 { code, data, msg
  }：

  - AppException（业务异常）→ 直接返回自定义 code/msg
  - HttpException（如验证失败）→ 包装为参数验证错误
  - 未处理异常 → 500 + "服务器内部错误"

  13. uuid — 会话/消息 ID 生成

  SessionService 中用 uuidv4() 生成唯一的会话 ID 和消息 ID。

  ---
  一句话总结技术协作流程：前端请求 → NestJS 路由 → class-validator 校验 →
  SessionService 管理会话 → AgentService 用 Vercel AI SDK 调用 LLM，LLM 通过 Zod
  定义的工具（TypeORM 查 MySQL / Redis 读缓存）自主获取数据 →
  结果通过流式响应返回前端。


  “Vercel AI SDK (ai v6)” 与 “@ai-sdk/openai-compatible” 两种技术的关系是什么？                  
  ⎿  ⧉ Selected 1 lines from docs\后端各技术的实际作用.md in Visual Studio Code

● 它们不是并列的技术，而是 主 SDK + 适配器（Provider） 的关系：

  ---
  Vercel AI SDK (ai v6) — 主 SDK

  这是核心库，提供 AI 能力的基础设施，比如：

  - generateText() — 发送消息并获取完整文本响应
  - streamText() — 发送消息并获取流式文本响应
  - tools() — 定义 LLM 可调用的工具
  - stepCountIs() — 控制工具调用的最大轮次

  独立使用它，只能连接 OpenAI 官方 API。

  ---
  @ai-sdk/openai-compatible — 适配器 Provider

  这是 Vercel AI SDK 的 官方 Provider 之一，作用是：让 AI SDK 能够调用 兼容 OpenAI API 格式的非
  OpenAI 模型。

  项目中用它连接智谱 GLM-4-Flash 和通义千问（通过阿里云百炼 DashScope），代码在 agent.service.ts:

  // 主 SDK 调用
  const result = await generateText({
      model: this.getModel(),  // ← 模型从适配器获取
      ...
  });

  // 适配器创建模型客户端（兼容 OpenAI 协议的非 OpenAI 服务）
  const client = createOpenAICompatible({
      name: 'llm-provider',
      apiKey: process.env.OPEN_API_KEY,
      baseURL: 'https://dashscope.aliyuncs.com/compatible-mode/v1',  // 阿里云百炼
  });
  const model = client('qwen-plus');

  ---
  类比理解

  把 Vercel AI SDK 想象成 统一插座，把 @ai-sdk/openai-compatible 想象成 转接头：

  ┌──────────────┬────────────────┬───────────────────────────────────────────────────┐
  │     关系     │      类比      │                     技术映射                      │
  ├──────────────┼────────────────┼───────────────────────────────────────────────────┤
  │ 统一接口     │ 插座标准化     │ ai SDK 提供 generateText/streamText 统一 API      │
  ├──────────────┼────────────────┼───────────────────────────────────────────────────┤
  │ 适配不同硬件 │ 不同插头要转接 │ 不同 LLM 服务（智谱、百炼、Anthropic）要 Provider │
  ├──────────────┼────────────────┼───────────────────────────────────────────────────┤
  │ 不用转接     │ 直接插         │ ai SDK 自带 @ai-sdk/openai，直连 OpenAI 官方 API  │
  └──────────────┴────────────────┴───────────────────────────────────────────────────┘

  项目中三个 Provider 的关系

  ┌───────────────────────────┬──────────────────────┬────────────────────────────────────────┐
  │            包             │         作用         │                  场景                  │
  ├───────────────────────────┼──────────────────────┼────────────────────────────────────────┤
  │ ai                        │ 主 SDK，提供核心 API │ 必须安装                               │
  ├───────────────────────────┼──────────────────────┼────────────────────────────────────────┤
  │ @ai-sdk/openai-compatible │ 适配器               │ 连接智谱、百炼等兼容 OpenAI 协议的模型 │
  ├───────────────────────────┼──────────────────────┼────────────────────────────────────────┤
  │ @ai-sdk/anthropic         │ 适配器               │ 连接 Anthropic Claude API              │
  └───────────────────────────┴──────────────────────┴────────────────────────────────────────┘

  配置里 LLM_PROVIDER 环境变量选择用哪个，当前默认用的是 @ai-sdk/openai-compatible 连接通义千问。
